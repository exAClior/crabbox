# 05 — Key pair management + user-data bootstrap

## Goal

Make sure the CVM instance starts with the right SSH key authorized, and the
right user-data running on first boot.

Lives in `worker/src/tencent.ts` next to the launch path; the helpers are
small and stay with the launch flow.

## Key pair lifecycle (mirror AWS `ensureSSHKey`)

Tencent CVM uses a **registered key pair object**, identified by `KeyId`
(`skey-...`), not just an inline `authorized_keys` line.

- Function: `ensureKeyPair(name, publicKey) → KeyId`.
  1. `DescribeKeyPairs` with `Filters=[{Name:"key-pair-name",Values:[name]}]`.
  2. If found, return its `KeyId`.
  3. Else `ImportKeyPair` with `KeyName=name`,
     `PublicKey=<openssh format string>`, `ProjectId=0`,
     `TagSpecification=[{ResourceType:"keypair", Tags:[...crabbox tags...]}]`.
- Function: `deleteSSHKey(name)`.
  1. `DescribeKeyPairs` to translate the name into a `KeyId`.
  2. `DeleteKeyPairs` with that `KeyId`.
  3. Treat "not found" as success (same as AWS `InvalidKeyPair.NotFound`).
- Naming: same scheme as AWS. The `config.providerKey` value generated by
  the coordinator is a `crabbox-<owner>-<random>` string; the
  `validTencentProviderKey` predicate matches that exact pattern so we never
  delete user-managed keys.

## RunInstances integration

`RunInstances.LoginSettings.KeyIds = [<KeyId>]`. The remote box ends up with
the public key in `/root/.ssh/authorized_keys` on first boot, and SSH access
is rejected with password.

## User-data

The Worker already has `awsUserData(config)` and `azureWindowsBootstrapPowerShell(config)`
in `worker/src/bootstrap.ts`. CVM accepts the same shape:

- **Linux**: cloud-init YAML or shell. Pass through `awsUserData(config)`
  base64-encoded as `RunInstances.UserData`. CVM's cloud-init is real
  cloud-init; the AWS bootstrap script should work unchanged. Confirm during
  smoke test.
- **Windows**: pass a `cloudbase-init` payload. CVM's Windows images ship
  with `cloudbase-init` enabled by default. The PowerShell content we use
  for Azure works there if wrapped as `<powershell>...</powershell>` per
  cloudbase-init's `Win32UserDataPlugin`. Build a small
  `tencentWindowsBootstrap(config)` that wraps the existing PowerShell.

The base64 wrap is the same: `btoa(string)`. Tencent docs cap user-data at
16 KB; that's the same as AWS, so the existing trimming code is fine.

## Tags on key pairs

Reuse `tencentTagsFromLabels(leaseProviderLabels(...))` but trim to just
`crabbox`, `created_by`, `owner`. Most label keys are lease-scoped and
shouldn't live on a key pair.

## Validation

Worker-side: when a request comes in with `provider=tencent` and no
`providerKey`, the coordinator generates one. The CVM client never
fabricates names — it only handles whatever name comes in.

## Open questions

- **Re-using user keys**: Tencent supports `Password` in `LoginSettings`
  instead of `KeyIds`. We won't expose this; password auth via the API is
  a footgun for ephemeral test boxes. Document the decision in the PR.
- **Key rotation**: out of scope. Each lease creates and deletes its own
  key pair, same as AWS.
- **Project IDs**: Tencent has a `ProjectId` field on most resources. We
  pin to `0` ("DEFAULT PROJECT") unless a future option exposes it.
